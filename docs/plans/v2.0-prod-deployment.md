## Database migrations (run in prod Supabase SQL Editor, IN ORDER)

All migration files are in `supabase/migrations/`. Only run the ones **not yet on prod**.

Prod currently has migrations through `20260322012736`. The following 8 migrations need to run:

| # | Migration file | What it does | Risk |
|---|---------------|-------------|------|
| 1 | `20260322200000_create-h2h-tables.sql` | H2H matches, match answers, player H2H stats tables + RLS | New tables only |
| 2 | `20260322210000_add-featured-badge-to-players.sql` | `featured_badge` column on players | Additive |
| 3 | `20260322220000_add-theme-to-players.sql` | `theme_id` column on players | Additive |
| 4 | `20260322230000_add-profile-social-fields.sql` | `featured_badges[]`, `bio`, `privacy_level` on players | Additive |
| 5 | `20260322240000_unique-callsign-constraint.sql` | Unique constraint on `display_name` | **Requires no dupes (step 1)** |
| 6 | `20260323300000_create-content-registry.sql` | 9 `registry_*` tables for admin content management | New tables only |
| 7 | `20260326000000_create-player-seen-moments.sql` | `player_seen_moments` table for SIGINT dialogue tracking | New table only |
| 8 | `20260327000000_create-promo-codes.sql` | Promo codes + redemptions tables (seed codes via dashboard) | New tables only |

**All migrations are additive** (new tables or new columns with defaults). Zero impact on existing v1.9 data. The only risky one is #5 (unique constraint) — verify no dupes first.

## Seed content registry (after migration #6)

```bash
SUPABASE_URL=<prod-url> SUPABASE_SERVICE_ROLE_KEY=<prod-service-role-key> npx tsx scripts/seed-content-registry.ts
```

This populates the registry tables with current game content (achievements, themes, quests, etc.). The admin UI at `/admin/content` reads from these tables.

## Rollback plan

If something breaks after deploy:

```bash
# Option A: Revert the merge commit on master
git revert HEAD
git push origin master

# Option B: Vercel instant rollback
# Dashboard → Deployments → promote previous deployment
```

DB migrations are additive (new tables/columns) — they don't need rollback. The old v1.9 code simply ignores the new columns and tables.
